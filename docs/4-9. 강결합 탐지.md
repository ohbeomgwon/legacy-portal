# 4-9. 강결합 탐지

서비스 계층이 협력 객체를 생성자 주입(DI) 없이 **직접 `new`로 생성**하는 지점을 찾는다. 이런 코드는
테스트에서 가짜(Fake/Mock) 구현으로 교체할 수 없고, 구현이 바뀌면 호출부를 전부 고쳐야 하는 강결합
(Tight Coupling) 상태다.

## 요약 표 — 직접 `new`로 생성되는 협력 객체(의존성)

| 파일 | 위치 | 생성 대상 | 인터페이스 존재 여부 |
|---|---|---|---|
| `ApprovalService.java` | L37 | `new SmtpMailSender()` | 없음(구체 클래스만 존재) |
| `ApprovalService.java` | L38 | `new FileAuditLogger()` | 없음(구체 클래스만 존재) |
| `NoticeService.java` | L24 | `new SmtpMailSender()` | 없음 |
| `NoticeService.java` | L25 | `new FileAuditLogger()` | 없음 |
| `ScheduleService.java` | L20 | `new FileAuditLogger()` | 없음 |

세 서비스 모두 필드 선언과 동시에 `= new ...()`로 초기화하고 있어, 생성자를 통해 다른 구현체(테스트용
스텁 등)를 주입할 방법이 없다. `ScheduleService`는 메일 발송 기능이 없어 `FileAuditLogger`만 해당된다.

## 상세

### `SmtpMailSender` (`common/SmtpMailSender.java`)

- 클래스 자체에 "구체 클래스를 서비스가 직접 new 한다(인터페이스 없음). 테스트에서 가짜로 교체 불가"라는
  스멜 주석이 이미 달려 있다(L5).
- 실제로는 `System.out.println`으로 콘솔에 출력만 하는 실습용 구현이며, `send(to, subject, body)` 메서드
  하나만 제공한다.
- 사용처: `ApprovalService`(L37, `processApproval()`의 상신/승인/반려 3곳에서 `mail.send(...)` 호출),
  `NoticeService`(L24, `publish()`의 긴급 공지 발송에서 호출).

### `FileAuditLogger` (`common/FileAuditLogger.java`)

- 마찬가지로 인터페이스 없는 구체 클래스이며, "출력 대상(파일/DB/콘솔)을 바꾸려면 호출부를 다 고쳐야
  한다"는 스멜 주석이 있다(L5).
- `write(line)` 메서드 하나만 제공하고, 실제로는 콘솔 출력만 한다(파일 append는 미구현).
- 사용처: `ApprovalService`(L38, `create()`와 `writeAudit()`에서 호출), `NoticeService`(L25,
  `create()`/`publish()`에서 호출), `ScheduleService`(L20, `create()`/`confirm()`에서 호출) — **세
  서비스 전부**가 의존한다.

## 참고 — 강결합에 해당하지 않는 `new` (구분을 위한 메모)

아래는 `new` 키워드가 쓰이지만 "의존성(협력 객체) 생성"이 아니라 **영속화할 도메인 엔티티 인스턴스를
만드는 것**이라 강결합 대상이 아니다. Repository를 통해 저장할 새 레코드를 표현하는 정상적인 객체
생성이며, DI로 대체할 개념이 아니다.

| 위치 | 코드 |
|---|---|
| `ApprovalService.create()` L48 | `Approval d = new Approval();` |
| `NoticeService.create()` L33 | `Notice n = new Notice();` |
| `ScheduleService.create()` L52 | `Schedule sc = new Schedule();` |
| `DataSeeder.run()` L24-27 | `new User(...)` × 4 (초기 데이터 시딩) |

## 강결합이 야기하는 관찰 가능한 문제

- **테스트 불가**: `ApprovalService`/`NoticeService`/`ScheduleService`를 단위 테스트하려면 실제
  `SmtpMailSender`/`FileAuditLogger`가 항상 함께 실행되어, 메일 발송·로그 기록을 가짜로 대체해 부작용
  없이 검증할 방법이 없다(현재 테스트 코드가 한 개도 없는 것과도 맞물린 문제).
- **구현 교체 불가**: 실제 SMTP 연동이나 파일/DB 감사 로그로 교체하려면 세 서비스의 필드 선언부를 모두
  찾아 고쳐야 한다(산탄총 수술).

## 리팩토링 방향 참고 (DIP 적용)

`CLAUDE.md`의 개선 방향(직접 `new` → 생성자 주입)에 따라 다음 순서를 제안한다.

1. `SmtpMailSender`, `FileAuditLogger`를 각각 인터페이스(`MailSender`, `AuditLogger` 등)로 추출하고,
   기존 구현 클래스는 그 인터페이스의 구현체로 유지한다(콘솔 출력 동작은 그대로 보존).
2. `ApprovalService`, `NoticeService`, `ScheduleService`의 생성자에 인터페이스 타입 파라미터를 추가해
   Spring이 빈을 주입하도록 바꾼다(필드의 `= new ...()` 초기화 제거).
3. `public` 메서드 시그니처(`processApproval`, `create`, `publish`, `confirm` 등)는 변경하지 않는다 —
   생성자 파라미터 추가는 API 계약이 아니므로 무방하지만, 서비스가 제공하는 메서드 이름·파라미터는
   `CLAUDE.md`의 계약 보존 규칙을 따른다.

리팩토링 후에도 감사 로그 문자열 형식, 메일 발송 시점·수신자·본문은 기존과 동일해야 하며, 특성화
테스트로 전후 동일성을 검증한다.
