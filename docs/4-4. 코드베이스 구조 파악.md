# 4-4. 코드베이스 구조 파악

KT DS 「클린코드 & 리팩토링 실전」 과정의 관통 실습 프로젝트인 **legacy-portal**(사내 업무 포털: 결재/공지/일정)의
현재 구조와 각 서비스의 책임을 정리한다. 이 문서는 리팩토링 실습을 시작하기 전, "레거시가 지금 무엇을 하고
어떤 스멜을 갖고 있는지"를 한눈에 보기 위한 지도(map)다.

## 1. 기술 스택 / 실행 방식

- Java 21 · Spring Boot 3.2.5 (Web, Data JPA) · JUnit 5(테스트 스타터만 포함, 실제 테스트 코드는 아직 없음)
- 빌드: Maven (`mvnw.cmd`)
- 실행: `.\mvnw.cmd spring-boot:run` (포트 8080)
- DB: `DataSourceConfig`가 기동 시 `localhost:3306/portal` MariaDB 접속을 2초 타임아웃으로 시도해서,
  되면 MariaDB, 안 되면 H2 인메모리(`jdbc:h2:mem:portal`)로 자동 폴백. 수강생 환경 세팅 부담을 없애기 위한 장치.
- `spring.jpa.hibernate.ddl-auto=create-drop` → 매 실행마다 스키마 재생성 + `DataSeeder`가 사용자 4명을 시드 데이터로 삽입.

## 2. 패키지 구조 개요

```
com.ktds.portal
├── PortalApplication      부트 진입점
├── DataSourceConfig       MariaDB/H2 자동 선택
├── DataSeeder             데모용 초기 사용자 4명 시드
├── approval/  (결재)      Approval, ApprovalController, ApprovalRepository, ApprovalService
├── notice/    (공지)      Notice, NoticeRepository, NoticeService  (Controller 없음)
├── schedule/  (일정)      Schedule, ScheduleRepository, ScheduleService  (Controller 없음)
├── user/      (사용자)    User, UserController, UserRepository
└── common/    (공통)      FileAuditLogger(감사로그), SmtpMailSender(메일)
```

각 도메인 폴더는 `Entity + Repository + Service (+ Controller)` 패턴을 따른다. `notice`·`schedule`은
아직 REST Controller가 없어 외부 API로 노출되어 있지 않다(서비스 계층까지만 존재).

## 3. 도메인별 책임

### 3.1 `approval` (결재) — 이 프로젝트의 핵심 실습 대상

- **Approval (Entity)**: 결재 문서. `status`(0=임시저장·1=상신·2=승인·3=반려·9=취소), `type`(1=지출·2=휴가·3=구매·4=기타),
  `priority`(1=낮음·2=보통·3=높음)가 모두 원시 `int`이고, 모든 필드에 public setter가 열려 있어 캡슐화가 없다.
- **ApprovalController**: `POST /api/approvals`(생성), `POST /api/approvals/{id}/process`(상신/승인/반려/취소 — action
  코드 1/2/3/9로 분기), `GET /api/approvals/drafts/{userId}`, `GET /api/approvals/inbox/{userId}`.
  요청 바디를 DTO 없이 `Map<String,Object>`로 받아 그대로 캐스팅해서 서비스로 흘려보낸다(검증 없음, 오탈자여도 컴파일은 통과).
- **ApprovalRepository**: `findByDrafterId`, `findByApproverId` — 단순 JPA 파생 쿼리.
- **ApprovalService**: 이 코드베이스의 "주인공 안티패턴" 클래스. 검증·영속화·메일 발송·감사 로그·상태 라벨링·금액 등급 계산·권한
  판정을 전부 혼자 처리하는 God Class다. 특히 `processApproval()` 하나가 상신/승인/반려/취소 네 가지 상태 전이를
  100줄 이상의 중첩 if로 처리하며, 승인 권한(`role >= 2`)이 없거나 상태가 안 맞으면 예외 없이 **조용히 무시**한다
  (레거시 관찰 동작 — 리팩토링 시 보존 대상). 금액 1,000,000원 이상 지출 결재는 상신 시 우선순위가 자동으로 "높음"으로
  올라가는 도메인 규칙도 서비스 안에 박혀 있다.

### 3.2 `notice` (공지)

- **Notice (Entity)**: `status`(0=임시·1=게시·9=내림), `category`(1=일반·2=긴급·3=인사), `pinned`(상단 고정 여부).
- **NoticeRepository**: 파생 쿼리 없이 `JpaRepository`만 상속.
- **NoticeService**: `create()`(작성)·`publish()`(게시, `role>=2` 팀장 이상만 가능)·`statusLabel()`(상태 라벨 변환)을 제공.
  `ApprovalService`와 거의 동일한 구조의 감사 로그·메일 발송·상태 라벨 로직이 복붙되어 있다 — 도메인 간 중복 제거
  실습(공통 모듈 추출)의 핵심 소재. 긴급(category=2) 공지를 게시하면 전체 사용자에게 메일을 보낸다.
- REST Controller가 없어 현재 API로는 노출되지 않는다.

### 3.3 `schedule` (일정)

- **Schedule (Entity)**: `status`(0=예정·1=확정·9=취소), `startAt`/`endAt`/`location`.
- **ScheduleRepository**: `findByOwnerId` 파생 쿼리.
- **ScheduleService**: `create()`(제목/시간 검증 + 같은 소유자의 기존 일정과 시간 겹침을 직접 루프로 검사 + 저장 + 감사 로그),
  `confirm()`(본인 소유 + 예정 상태일 때만 확정). 겹침 검사 로직이 `flag1`이라는 의미 불명 변수와 함께 인라인으로 박혀 있어
  가독성이 낮고, 검증 실패 시 원인 구분 없이 전부 `null`을 반환한다.
- 역시 REST Controller 없음.

### 3.4 `user` (사용자)

- **User (Entity)**: `role`(1=사원·2=팀장·3=임원)이 원시 `int`. `approval`/`notice` 서비스의 권한 판정(`role>=2`)이
  이 값에 의존한다.
- **UserController**: `GET /api/users` — 데모 프론트엔드의 로그인(사용자 선택) 화면용 단일 엔드포인트.
- **UserRepository**: 파생 쿼리 없이 `JpaRepository`만 상속.

### 3.5 `common` (공통 인프라)

- **FileAuditLogger**: 감사 로그 기록기. 실제로는 콘솔 출력만 하며(파일 append는 미구현), 인터페이스 없는 구체
  클래스라 각 서비스가 `new FileAuditLogger()`로 직접 생성해서 쓴다.
- **SmtpMailSender**: 메일 발송기. 역시 콘솔 출력만 하는 구체 클래스이며 인터페이스가 없어 서비스들이 직접 `new`한다.

### 3.6 애플리케이션 부트스트랩

- **PortalApplication**: `@SpringBootApplication` 진입점.
- **DataSourceConfig**: MariaDB 연결 가능 여부를 확인해 MariaDB/H2 중 `DataSource`를 선택하는 `@Configuration`.
- **DataSeeder**: `CommandLineRunner`로 기동 시 사용자가 없으면 데모용 사용자 4명(사원 2, 팀장 1, 임원 1)을 시드.

## 4. 도메인 간 공통적으로 관찰되는 구조적 특징

이 특징들은 전부 **의도적으로 심어둔 스멜**이며, 리팩토링 실습 단계에서 관찰 가능한 동작(계약·DB 저장값·상태 전이
결과)은 그대로 보존한 채 내부 구조만 개선하는 대상이다.

| 스멜 | 나타나는 곳 |
|---|---|
| God Class (검증+영속화+메일+로그+포맷팅+권한판정을 한 클래스가 전담) | `ApprovalService` |
| Long Method (긴 메서드, 깊은 중첩 if) | `ApprovalService.processApproval()`, `ScheduleService.create()` |
| Magic Number / Primitive Obsession (status·type·role·priority·category가 전부 int) | `Approval`, `Notice`, `Schedule`, `User` 전체 |
| Duplicated Code (감사 로그 기록, 메일 본문 생성, 상태 라벨 변환이 서비스마다 복붙) | `ApprovalService` ↔ `NoticeService` ↔ `ScheduleService` |
| Tight Coupling (협력 객체를 생성자 주입 없이 직접 `new`) | `ApprovalService`, `NoticeService`, `ScheduleService` → `SmtpMailSender`, `FileAuditLogger` |
| Feature Envy (엔티티 필드를 꺼내 서비스가 직접 도메인 규칙을 계산) | `ApprovalService.amountGrade()`, 금액 기준 우선순위 자동 상향 |
| Poor Naming (약어 변수: `d`, `u`, `s`, `proc`, `tmp`, `flag1`) | `ApprovalService`, `ScheduleService` |
| 캡슐화 부재 (모든 엔티티 필드에 public setter) | `Approval`, `Notice`, `Schedule`, `User` |
| 요청 검증 부재 (Controller가 `Map<String,Object>`를 그대로 캐스팅) | `ApprovalController` |
| 조용한 실패 (권한 없음/상태 불일치 시 예외 없이 무시하거나 `null` 반환) | `ApprovalService.processApproval()`, `ScheduleService.create()` |
| 테스트 부재 (특성화 테스트 없음 — 안전망 없이 리팩토링 위험) | 전체 |

## 5. 리팩토링 우선순위 참고

`CLAUDE.md`의 개선 방향(매직넘버→enum, `new`→생성자 주입, Entity 직접 노출→DTO 분리, 긴 메서드 분리·중복 제거)을
적용할 때 가장 먼저 손댈 곳은 중복과 결합도가 집중된 `ApprovalService`/`NoticeService`/`ScheduleService`의
감사 로그·메일 발송·상태 라벨 로직이며, 그 다음이 `Approval.status`/`type`/`priority`, `Notice.status`/`category`,
`Schedule.status`, `User.role`의 enum화(단, DB에는 기존 정수 그대로 저장 — `@Convert` + `AttributeConverter` 사용,
`@Enumerated` 금지)다. 단, 모든 변경은 특성화 테스트가 리팩토링 전후 동일하게 green인 것으로 검증해야 한다.
