# 4-13. processApproval 상태 전이 규칙

`ApprovalService.processApproval(id, userId, action, reason)`(L75-160)의 현재 상태 전이 규칙을
action별로 정리한다. God Method 분해(`docs/4-12` BL-05) 전, 리팩토링 대상 동작을 명세로 고정해
`docs/4-13-테스트`(특성화 테스트 6개, `ApprovalServiceProcessApprovalCharacterizationTest`)와
상호 검증하기 위한 문서다.

## 공통 전제조건 (모든 action에 선행)

| 조건 | 불충족 시 동작 |
|---|---|
| `id`로 조회한 `Approval`이 존재해야 함 | 조용히 `return` (L77-79) |
| `userId`로 조회한 `User`가 존재해야 함 | 조용히 `return` (L82-84) |
| `action`이 1/2/3/9 중 하나여야 함 | 위 값이 아니면 `if-else if` 체인 어디에도 걸리지 않아 **아무 일도 일어나지 않음**(L90-159) |

## action별 상태 전이 표

| action | 처리명 | 전제조건 (현재 status) | 전제조건 (사용자) | 성공 시 결과 status | 부가 효과 (성공 시) | 조건 불충족 시 |
|---|---|---|---|---|---|---|
| **1** | 상신 (SUBMIT) | `status == 0`(임시저장) | 없음(기안자·결재자 확인 안 함) | `1`(상신) | `type==1`(지출) && `amount>=1,000,000`이면 `priority=3`(높음) 자동 상향 → 결재자에게 메일 → 감사로그 SUBMIT | 상태가 0이 아니면 조용히 무시 (L90-109) |
| **2** | 승인 (APPROVE) | `status == 1`(상신) | `approverId == userId` **그리고** `user.role >= 2`(팀장 이상) | `2`(승인) | 기안자에게 메일 → 감사로그 APPROVE | 상태 불일치·본인이 결재자가 아님·권한 부족 중 하나라도 해당하면 조용히 무시 (L110-128) |
| **3** | 반려 (REJECT) | `status == 1`(상신) | `approverId == userId` **그리고** `user.role >= 2`(팀장 이상) | `3`(반려) | `rejectReason` 저장 → 기안자에게 메일(사유 포함) → 감사로그 REJECT | 승인과 동일 조건, 불충족 시 조용히 무시 (L129-148) |
| **9** | 취소 (CANCEL) | `status == 0`(임시저장) **또는** `status == 1`(상신) | `drafterId == userId` (기안자 본인만) | `9`(취소) | 메일 없음 → 감사로그 CANCEL만 | 상태 불일치 또는 본인이 기안자가 아니면 조용히 무시 (L149-159) |

## 특기사항

- **승인(2)과 반려(3)의 전제조건이 완전히 동일**하다(상태=상신, 결재자 본인, role≥2) — `docs/4-5`의
  중복 코드 E항목과 같은 지점.
- **취소(9)만 권한 판정이 다르다** — role 체크 없이 "기안자 본인"만 확인하고, 승인 후(`status==2`)에는
  취소가 불가능하다(0 또는 1 상태만 허용).
- **승인/반려에는 없고 상신에만 있는 것**: 금액 기준 우선순위 자동 상향 규칙(L94-96).
- 모든 "조건 불충족" 경로는 예외 없이 조용히 무시되며, 호출자(컨트롤러/프론트엔드)는 성공·실패를
  구분할 방법이 없다 — `docs/4-4`·`4-10`에서 "조용한 실패"로 이미 진단한 부분과 일치하고, `CLAUDE.md`
  규칙상 리팩토링 단계에서는 이 동작을 그대로 보존해야 한다.

## 검증 — 특성화 테스트와의 대응

이 표의 각 행은 `src/test/java/com/ktds/portal/approval/ApprovalServiceProcessApprovalCharacterizationTest.java`의
아래 테스트로 실제 실행을 통해 검증했다(`mvn test` 기준 6/6 통과).

| 표의 규칙 | 대응 테스트 메서드 |
|---|---|
| action=1→2 성공 전이 (상신→승인) | `상신후_승인하면_승인상태가_된다` |
| action=3 성공 전이 + rejectReason 저장 | `상신후_반려하면_반려상태가_되고_사유가_저장된다` |
| action=9 성공 전이 (임시저장→취소) | `임시저장_상태에서_취소하면_취소상태가_된다` |
| action=2 실패 — role<2 (권한 없음) | `권한없는_결재자가_승인시도하면_무시된다` |
| 공통 전제조건 실패 — 존재하지 않는 id | `존재하지_않는_id로_처리하면_예외없이_무시된다` |
| action=2 실패 — 이미 승인된 상태(status≠1)에서 재승인 | `이미_승인된_건을_재승인하면_무시된다` |

리팩토링(God Method 분해 등) 이후에도 이 표와 특성화 테스트 6개가 변경 없이 일치해야 "동작을
바꾸지 않았다"고 확인할 수 있다.
