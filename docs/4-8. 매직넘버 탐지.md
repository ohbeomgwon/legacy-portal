# 4-8. 매직넘버 탐지

`approval`(결재) · `notice`(공지) · `schedule`(일정) · `user`(사용자) 네 도메인에 흩어진 매직넘버를
필드별로 모으고, 코드 주석과 실제 사용 위치를 근거로 의미를 정리한다. 모두 `int` 필드/파라미터이며,
의미는 코드가 아니라 주석에만 존재한다는 것이 공통된 문제다.

## 2. `Approval` 처리 액션 파라미터 (`action`/`proc`) — 1 / 2 / 3 / 9

`ApprovalController.process()`가 받는 `action`과 `ApprovalService.processApproval()`의 `proc`은
**같은 숫자 체계를 상태값과 공유**한다("상신"=1, "승인"=2, "반려"=3, "취소"=9로 상태값과 1:1 대응).

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 1 | 상신 처리 | `ApprovalController.java` L36,42 주석 / `ApprovalService.java` L73,90 |
| 2 | 승인 처리 | L110 |
| 3 | 반려 처리 | L129 |
| 9 | 취소 처리 (4~8은 정의되지 않은 결번) | L149 — 코드 주석: "왜 9? 4~8 은 비워둔 규칙 없는 번호" |

API 요청 바디의 `action` 필드가 이 정수를 그대로 요구해, 프론트엔드가 숫자 뜻을 외워야 하는 계약으로
노출되어 있다(`ApprovalController.java` L36-45).

## 5. 금액 관련 임계값 — 100,000 / 1,000,000 / 10,000,000

`ApprovalService.amountGrade()`(L182-187)에서 등급을 매기는 기준.

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 1,000,000 이상 | 결재 상신 시 우선순위를 "높음"으로 자동 상향하는 기준(지출 결재 한정) | L94 |
| 10,000,000 이상 | 금액 등급 "S" | L184 주석: "1000만원=S" |
| 1,000,000 이상 | 금액 등급 "A" | L185 주석: "100만원=A" |
| 100,000 이상 | 금액 등급 "B" | L186 주석: "10만원=B" |
| 그 외 | 금액 등급 "C" | L187 |

`1,000,000`이 "우선순위 자동 상향 기준"과 "금액 등급 A 기준" 두 곳에 각각 독립적으로 하드코딩되어
있어, 같은 숫자라도 우연의 일치인지 의도된 동일 기준인지 코드만으로는 알 수 없다.

## 11. 같은 숫자, 다른 의미 — 컨텍스트 의존 위험

아래처럼 필드마다 같은 정수 리터럴이 전혀 다른 의미로 재사용되고 있어, 코드를 읽을 때 "이 필드가
무엇인지"를 모르면 숫자만으로 의미를 알 수 없다.

| 값 | Approval.status/action | Approval.type | Approval.priority | Notice.status | Notice.category | Schedule.status | User.role |
|---|---|---|---|---|---|---|---|
| 0 | 임시저장 | — | — | 임시 | — | 예정 | — |
| 1 | 상신 | 지출 | 낮음 | 게시 | 일반 | 확정 | 사원 |
| 2 | 승인 | 휴가 | 보통 | — | 긴급 | — | 팀장 |
| 3 | 반려 | 구매 | 높음 | — | 인사 | — | 임원 |
| 9 | 취소 | — | — | 내림 | — | 취소 | — |

한 가지 일관된 패턴은 있다 — **`0`은 세 도메인 모두에서 "초기/대기 상태"(임시저장·임시·예정)**,
**`9`는 세 도메인 모두에서 "종료/취소성 상태"(취소·내림·취소)**로 쓰인다. 반면 `1`/`2`/`3`은 필드마다
의미가 완전히 달라 이 규칙이 없고, 이 비일관성이 가장 큰 오독 위험이다.

## 12. Enum 리팩토링 후보 정리

`CLAUDE.md` 규칙에 따라 아래 열거형 후보들은 **DB에는 기존 정수를 그대로 저장**해야 하므로
`@Convert` + `AttributeConverter`로 매핑하고, `@Enumerated(STRING/ORDINAL)`은 사용하지 않는다.

| 도메인 필드 | Enum 후보 이름(예) | 코드값 매핑 |
|---|---|---|
| `Approval.status` / action 파라미터 | `ApprovalStatus` | DRAFT=0, SUBMITTED=1, APPROVED=2, REJECTED=3, CANCELED=9 |
| `Approval.type` | `ApprovalType` | EXPENSE=1, LEAVE=2, PURCHASE=3, ETC=4 |
| `Approval.priority` | `ApprovalPriority` | LOW=1, NORMAL=2, HIGH=3 |
| `Notice.status` | `NoticeStatus` | DRAFT=0, PUBLISHED=1, TAKEN_DOWN=9 |
| `Notice.category` | `NoticeCategory` | GENERAL=1, URGENT=2, HR=3 |
| `Schedule.status` | `ScheduleStatus` | PLANNED=0, CONFIRMED=1, CANCELED=9 |
| `User.role` | `UserRole` | STAFF=1, MANAGER=2, EXECUTIVE=3 |

`role >= 2`("팀장 이상") 같은 대소비교 판정은 enum 전환 시 순서(ordinal)에 의존하지 않고
`UserRole.canApprove()`처럼 도메인 메서드로 옮기는 것이 안전하다 — DB 저장값은 정수로 고정되지만
enum의 선언 순서가 바뀌면 `ordinal()` 기반 비교는 깨질 수 있기 때문이다.

## 13. 검증

모든 값과 근거는 현재 소스 코드의 주석·조건식을 그대로 인용한 것이며, enum화 시에도 `CLAUDE.md`의
불변 규칙(DB 저장값 변경 금지, API·계약 변경 금지)에 따라 이 표의 정수 매핑이 그대로 유지되어야 하고,
특성화 테스트로 리팩토링 전후 동일 동작을 검증한다.
