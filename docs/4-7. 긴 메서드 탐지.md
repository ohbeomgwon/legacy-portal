# 4-7. 긴 메서드 탐지

`approval`(결재) · `notice`(공지) · `schedule`(일정) 세 서비스에 존재하는 모든 메서드의 **줄 수**와
**최대 중첩 깊이**를 측정해 표로 정리한다. 리팩토링 우선순위를 정하기 위한 "긴 메서드(Long Method)"
스멜의 실측 자료다.

## 측정 기준

- **줄 수**: 메서드 시그니처 줄부터 닫는 중괄호(`}`) 줄까지 포함한 총 라인 수(주석·빈 줄 포함).
- **중첩 깊이**: 메서드 본문을 깊이 1로 두고, `if`/`else if`/`for`/`while` 블록이 하나 열릴 때마다 +1.
  중괄호 없는 단문 `if`(예: `if (s == 0) tmp = "...";`)도 조건 분기 한 단계로 세어 깊이 2로 계산한다.
- **50줄 초과** 여부와 **중첩 깊이 4단계 이상**(가독성이 급격히 떨어지기 시작하는 경계)을 함께 표시했다.

**50줄을 초과하는 메서드는 `ApprovalService.processApproval()` 단 하나**이며, 중첩 깊이도 표 전체에서
가장 깊다(6단계). `NoticeService.publish()`(25줄)·`ScheduleService.create()`(39줄)는 50줄 미만이라
"긴 메서드" 기준에는 안 걸리지만, 각각 5단계·4단계 중첩으로 **줄 수 대비 복잡도가 과도하게 높은
케이스**라 별도로 표시했다.

## 상세: `ApprovalService.processApproval()` — 6단계 중첩 구조

```
depth1  method body
  ├─ if (d == null) return;                                         depth2 (조회 실패 가드)
  ├─ if (u == null) return;                                         depth2 (조회 실패 가드)
  └─ if (proc == 1) {                    ← 상신                      depth2
        if (s == 0) {                                                depth3
          if (type==1 && amount>=1000000) { ... }                    depth4
          if (approver != null) { ... }                              depth4
        }
     } else if (proc == 2) {             ← 승인                      depth2
        if (s == 1) {                                                depth3
          if (approverId.equals(userId)) {                           depth4
            if (u.getRole() >= 2) {                                  depth5
              if (drafter != null) { ... }                           depth6  ← 최대 깊이
            }
          }
        }
     } else if (proc == 3) {             ← 반려 (승인과 동일 구조 복붙) depth2
        if (s == 1) {                                                depth3
          if (approverId.equals(userId)) {                           depth4
            if (u.getRole() >= 2) {                                  depth5
              if (drafter != null) { ... }                           depth6  ← 최대 깊이
            }
          }
        }
     } else if (proc == 9) {             ← 취소                      depth2
        if (s == 0 || s == 1) {                                      depth3
          if (drafterId.equals(userId)) { ... }                      depth4
        }
     }
```

승인(proc==2)과 반려(proc==3) 분기가 `상태 확인 → 결재자 본인 확인 → 권한(role≥2) 확인 → 알림 대상
존재 확인`이라는 동일한 4중 중첩 구조를 그대로 복붙하고 있어, 줄 수뿐 아니라 **중첩 구조 자체의
중복**도 함께 나타난다(`docs/4-5. 중복 코드 탐지.md`의 D·E 항목과 겹치는 지점).

## 참고: 짧지만 중첩이 깊은 메서드

- **`NoticeService.publish()`(25줄, 깊이 5)**: `역할 권한 → 상태 확인 → 긴급 분류 확인 → 전체 사용자
  반복문`이 한 메서드 안에 순차적으로 중첩되어 있다. 줄 수 기준(50줄)만 보면 놓치기 쉬운 케이스라
  별도 주의가 필요하다.
- **`ScheduleService.create()`(39줄, 깊이 4)**: 앞부분 3개의 단문 유효성 검사(guard)에 이어, 시간
  겹침을 검사하는 `for` 안에 `if`가 두 겹 중첩된 루프가 이어진다. 검증 로직과 겹침 판정 로직을
  분리하면 깊이를 자연스럽게 낮출 수 있는 후보다.

## 리팩토링 우선순위 참고

1. **`ApprovalService.processApproval()`** — 최우선 대상. `submit()`/`approve()`/`reject()`/`cancel()`
   4개 메서드로 추출하고, `processApproval(id, userId, action, reason)` 자체는 시그니처를 유지한 채
   내부에서 action 값에 따라 위임만 하도록 바꾼다(`CLAUDE.md` public 메서드 계약 보존 규칙).
2. **`NoticeService.publish()`** — 권한 판정·상태 확인·긴급 메일 발송을 별도 메서드로 추출해 깊이를
   낮춘다.
3. **`ScheduleService.create()`** — 유효성 검사(가드 절)와 시간 겹침 판정(`isOverlapping` 같은 이름의
   메서드로 추출)을 분리한다.

모든 추출은 상태 전이 결과·응답·DB 저장값이 리팩토링 전후 동일한지 특성화 테스트로 검증한 뒤 진행한다.
