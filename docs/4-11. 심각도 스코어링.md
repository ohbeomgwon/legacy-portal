# 4-11. 심각도 스코어링

`docs/4-10. 도출 스멜 - 심각도 표.md`가 "지금 당장 얼마나 위험한가"를 기준으로 서열을 매겼다면, 이
문서는 각 스멜을 **영향도 × 수정난이도**로 채점해 별점(★)으로 환산한다. 목적이 다르므로 두 문서의
순서가 완전히 같지는 않다 — 이 차이 자체가 착수 순서를 정하는 데 중요한 정보다(문서 하단 참고).

## 채점 기준

### 영향도 (Impact, 1~5)

| 점수 | 기준 |
|---|---|
| 5 | 전체 코드베이스/모든 리팩토링의 안전성에 영향을 주는 구조적 근본 원인 |
| 4 | 여러 도메인에 걸친 광범위한 영향, 또는 API 경계에서 외부 입력에 노출되는 위험 |
| 3 | 특정 서비스·클래스 내에 국한되지만 유지보수 리스크가 뚜렷함 |
| 2 | 가독성·스타일 저하 — 동작에는 영향 없음 |
| 1 | 거의 영향 없는 사소한 문제 |

### 수정난이도 (Difficulty, 1~5)

| 점수 | 기준 |
|---|---|
| 5 | `CLAUDE.md` 불변 규칙상 **리팩토링 단계에서 손댈 수 없음**(고치면 관찰 가능한 동작이 바뀌어 리팩토링이 아니라 재설계가 됨) 또는 초광범위 구조 변경 필요 |
| 4 | 여러 클래스·도메인에 걸친 구조 변경 필요(추출 + 주입 + 회귀 검증 동반) |
| 3 | 단일 클래스 내 구조 변경(메서드 분리, 책임 이동) 필요 |
| 2 | 기계적 치환(이름 변경, enum 매핑, 인터페이스 추출)으로 비교적 안전하게 가능 |
| 1 | 국소적 수정으로 즉시 가능 |

### 점수 → 별점 환산

심각도 점수 = 영향도 × 수정난이도 (최대 5×5=25). 아래 구간으로 별점을 매긴다.

| 점수 구간 | 별점 |
|---|---|
| 21~25 | ★★★★★ |
| 16~20 | ★★★★ |
| 11~15 | ★★★ |
| 6~10 | ★★ |
| 1~5 | ★ |

## 참고 — 점수가 높다고 먼저 고치는 것이 아니다 (사분면 해석)

영향도×난이도 점수는 "위험 노출도"이지 "착수 순서"가 아니다. 같은 점수라도 **난이도가 낮은 쪽을 먼저**
처리하는 것이 정석이다(퀵윈 우선). 네 사분면으로 나누면 다음과 같다.

| | 난이도 낮음 (1~2) | 난이도 높음 (3~5) |
|---|---|---|
| **영향도 높음 (4~5)** | 🟢 Quick Win — 먼저 착수: 테스트 부재, Magic Number | 🔴 Major Project — 계획적 투자, 테스트 선행 필수: God Class, 조용한 실패, Long Method, 요청 검증 부재 |
| **영향도 낮음 (1~3)** | 🟡 Fill-in — 여유 있을 때 정리: Poor Naming, Comment Smell, Long Parameter List, Tight Coupling | ⚪ Thankless — 지금은 손대지 않음: 비대칭/미구현 매직값 |

- **Quick Win**(테스트 부재, Magic Number)은 점수가 중간(10, 8)이라도 난이도가 낮아 가장 먼저 처리해야
  한다 — 특히 테스트 부재는 아래 Major Project 항목을 안전하게 진행하기 위한 전제 조건이라 사실상
  순서상 0순위다.
- **Major Project**(God Class, 조용한 실패, Long Method, 요청 검증 부재)는 점수·별점이 높은데, 여러
  클래스에 걸친 구조 변경이 필요하거나(God Class·Long Method·요청 검증) 아예 리팩토링 범위 밖의
  결정이 필요하다(조용한 실패). 반드시 테스트가 먼저 갖춰진 뒤 손댄다.
- **Fill-in**은 점수가 낮아 급하지 않지만 난이도도 낮아 여유 시간에 처리하면 이득이 큰 항목이다.
- **Thankless**(비대칭/미구현 매직값)는 영향도가 낮으면서 "고치는 것 자체가 리팩토링 범위를 벗어나는"
  항목이라 지금 단계에서는 의도적으로 미룬다.

## 특기사항 — 왜 "조용한 실패"·"비대칭/미구현 매직값"의 수정난이도가 5인가

두 항목은 기술적으로 고치기 어려운 게 아니라, **`CLAUDE.md`의 불변 규칙상 리팩토링 단계에서 고치는
행위 자체가 금지**되어 있어서 최고 난이도를 부여했다.

> 레거시의 나쁜 설계(예: 권한 없는 승인을 조용히 무시)도 리팩토링 단계에선 그대로 보존한다 ... 결함을
> 바로 고치면 리팩토링이 아니라 동작 변경이다 → 리팩토링 완료 후 별도 단계(재설계)에서 다룬다.

즉 "조용한 실패를 예외로 바꾸기"·"`Notice.category=3`(인사) 분기 구현하기" 같은 수정은 지금 시도하는
순간 리팩토링의 정의를 벗어난다. 이 두 항목의 난이도 5는 "지금은 만지지 말라"는 신호로 읽어야 한다.

## 결론 — 실행 순서 제안

1. **Quick Win 먼저**: 특성화 테스트 작성(테스트 부재) → 매직넘버 enum화(`@Convert` 방식) → 강결합
   해소(인터페이스 추출 + 생성자 주입)
2. **Major Project는 테스트 확보 후**: God Class(`processApproval` 분해) → Long Method/깊은 중첩 해소
   → 요청 검증(DTO record 도입)
3. **Fill-in은 짬날 때**: 네이밍 개선, 파라미터 객체화, 캡슐화 강화, Feature Envy 이동
4. **Thankless는 이번 리팩토링 범위에서 제외**: 조용한 실패·비대칭 매직값은 관찰 동작으로 보존하고,
   차후 별도 재설계 단계 백로그로만 남긴다.

모든 단계의 완료 판정은 특성화 테스트가 리팩토링 전후 동일하게 green인지로 검증한다.
