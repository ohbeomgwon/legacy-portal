# 4-12. 리팩토링 백로그

`docs/smells.md`(종합 스멜 목록)와 `docs/4-10`(심각도 서열)·`docs/4-11`(영향도×난이도 점수, 사분면
해석)을 근거로 실행 가능한 백로그를 우선순위 순으로 정리한다. `4-11`의 결론을 그대로 순서의 뼈대로
삼되, 서로 맞물린 스멜은 하나의 백로그 항목으로 묶었다.

## 백로그 요약 표

| 순번 | 백로그 항목 | 관련 스멜 (smells.md) | 우선순위 그룹 | 선행 조건 |
|---|---|---|---|---|
| BL-01 | 특성화 테스트 작성 | 1. 테스트 부재 | 🟢 Quick Win (0순위) | 없음 |
| BL-02 | 매직넘버 → enum 치환 (`@Convert`) | 4. Magic Number/Primitive Obsession | 🟢 Quick Win | BL-01 |
| BL-03 | 공통 인프라 인터페이스 추출 + 생성자 주입 | 6. Tight Coupling | 🟢 Quick Win | BL-01 |
| BL-04 | 서비스 간 중복 코드 제거 | 5. Duplicated Code (A~F), 13. 조용한 실패 중 D 패턴만* | 🟢 Quick Win | BL-02, BL-03 |
| BL-05 | God Class 분해 (`processApproval` 내부 위임) | 2. God Class, 3. Long Method·깊은 중첩 | 🔴 Major Project | BL-01~04 |
| BL-06 | 요청 DTO(record) 도입 | 14. 요청 검증 부재 | 🔴 Major Project | BL-01 (BL-05와 병행 가능) |
| BL-07 | 캡슐화 강화 + Feature Envy 이동 | 7. 캡슐화 부재, 8. Feature Envy | 🟡 Fill-in | BL-05 |
| BL-08 | 파라미터 목록 정리 + 네이밍 개선 | 9. Long Parameter List, 10. Poor Naming, 12. Comment Smell | 🟡 Fill-in | BL-06 |
| BL-09 | (보류) 조용한 실패 개선 — 재설계 백로그로 이관 | 13. 조용한 실패 | ⚪ Thankless — 이번 범위 제외 | 해당 없음 |
| BL-10 | (보류) 비대칭/미구현 매직값 정리 — 재설계 백로그로 이관 | 11. 비대칭/미구현 매직값 | ⚪ Thankless — 이번 범위 제외 | 해당 없음 |

\* BL-04의 "D. 조회+null 체크 후 조용히 리턴" 패턴은 **구조(중복된 코드 모양)만 공통 메서드로 추출**하고,
"조회 실패 시 조용히 리턴한다"는 **동작 자체는 그대로 보존**한다. 동작을 바꾸는 것은 BL-09(보류)의 몫이다.

## 상세

### BL-01 — 특성화 테스트 작성 (선행 필수, Quick Win)

- **왜 0순위인가**: `4-11`에서 영향도 5·난이도 2(점수 10, ★★)로 산출됐지만, 아래 모든 항목의 안전망이라
  실질적으로 순서상 가장 먼저다.
- **범위**: `ApprovalService`(`create`, `processApproval` 4개 전이 경로, `statusLabel`, `amountGrade`),
  `NoticeService`(`create`, `publish`), `ScheduleService`(`create`, `confirm`) — happy path뿐 아니라
  **조용한 실패 케이스**(대상 없음, 권한 없음, 상태 불일치, 시간 겹침)까지 포함해 현재 동작을 고정한다.
- **완료 조건(DoD)**: 주요 메서드별 성공/실패 케이스 테스트가 green, DB 저장값·응답·상태 라벨이 현재
  값과 일치.

### BL-02 — 매직넘버 → enum 치환

- **관련 스멜**: `smells.md` 4번. `Approval.status`/`type`/`priority`, `Notice.status`/`category`,
  `Schedule.status`, `User.role`을 enum화하되 **DB 저장값은 정수 그대로**(`@Convert` +
  `AttributeConverter`, `@Enumerated` 금지).
- **매핑표**: `docs/4-8. 매직넘버 탐지.md` 12절의 표를 그대로 사용.
- **주의**: `role >= 2` 같은 대소비교는 enum `ordinal()`에 의존하지 말고 `UserRole.canApprove()` 같은
  도메인 메서드로 옮긴다(`4-8` 12절 참고).
- **DoD**: enum 도입 후 BL-01 테스트가 변경 없이 green, DB 컬럼 값(정수)이 리팩토링 전후 동일.

### BL-03 — 공통 인프라 인터페이스 추출 + 생성자 주입

- **관련 스멜**: `smells.md` 6번. `SmtpMailSender`→`MailSender`, `FileAuditLogger`→`AuditLogger`
  인터페이스 추출 후 세 서비스 생성자에 주입.
- **DoD**: 서비스 필드에 `new SmtpMailSender()`/`new FileAuditLogger()` 없음, 콘솔 출력 형식·시점은
  기존과 동일, BL-01 테스트 green.

### BL-04 — 서비스 간 중복 코드 제거

- **관련 스멜**: `smells.md` 5번 A~F 전부(감사 로그 포맷, 메일 본문 생성, 조회+null 체크 구조, 권한
  판정 조건식, 상태 라벨 변환).
- **의존**: BL-02(enum)·BL-03(인터페이스)이 먼저 있어야 상태 라벨(F)과 권한 판정(E)을 자연스럽게
  enum/도메인 메서드로 통합할 수 있다.
- **DoD**: `ApprovalService`/`NoticeService`/`ScheduleService`에 동일 블록 복붙이 남지 않음, `role>=2`
  조건식이 한 곳(도메인 메서드)에서만 정의됨, BL-01 테스트 green.

### BL-05 — God Class 분해 (Major Project)

- **관련 스멜**: `smells.md` 2, 3번. `ApprovalService.processApproval()`(86줄, 중첩 6단계)을
  `submit()`/`approve()`/`reject()`/`cancel()`로 내부 위임 분리.
- **불변 규칙**: `public void processApproval(Long id, Long userId, int action, String reason)`
  시그니처는 그대로 유지하고, 내부에서 action 값에 따라 위임만 한다(`CLAUDE.md` public 메서드 계약
  보존).
- **의존**: BL-01~04가 먼저 끝나 있어야 위험 없이 진행 가능(`4-11`에서 난이도 4로 산정한 이유).
- **DoD**: 각 위임 메서드가 50줄 이내·중첩 3단계 이내, BL-01 테스트 green, 상태 전이 결과·메일·감사
  로그가 기존과 동일.

### BL-06 — 요청 DTO(record) 도입

- **관련 스멜**: `smells.md` 14번. `ApprovalController`의 `Map<String,Object>` 캐스팅을 request
  record로 교체.
- **불변 규칙**: JSON 필드명·구조는 레거시와 100% 동일하게 유지(`CLAUDE.md` 요청/응답 형식 변경 금지).
- **DoD**: 컨트롤러가 타입 안전 DTO를 사용, 동일한 JSON 요청에 대해 동일한 응답, BL-01 테스트 green.

### BL-07 — 캡슐화 강화 + Feature Envy 이동 (Fill-in)

- **관련 스멜**: `smells.md` 7, 8번. `amountGrade()`·금액 기준 우선순위 자동 상향 로직을 `Approval`
  도메인으로 이동, public setter 노출을 최소화.
- **의존**: BL-05로 서비스 구조가 정리된 뒤 진행해야 중복 작업이 없다.
- **DoD**: 도메인 규칙이 `Approval` 엔티티(또는 도메인 메서드)에 위치, 서비스는 위임만 함.

### BL-08 — 파라미터 목록 정리 + 네이밍 개선 (Fill-in)

- **관련 스멜**: `smells.md` 9, 10, 12번. `create()` 8개 파라미터를 BL-06에서 만든 요청 DTO로 대체,
  `d`/`u`/`s`/`proc`/`tmp`/`flag1` 등 약어 변수를 의미 있는 이름으로 개선(레거시 약어를 그대로 옮기지
  않음 — `CLAUDE.md` 네이밍 규칙).
- **DoD**: 파라미터 목록이 DTO 하나로 축소, 약어 변수 제거, 나쁜 이름을 변명하던 주석도 함께 제거.

### BL-09, BL-10 — 이번 리팩토링 범위에서 제외 (보류)

`4-11`에서 수정난이도 5(리팩토링 단계에서 손댈 수 없음)로 산정한 두 항목은 이번 백로그에 포함하지
않는다. "조용한 실패를 예외로 바꾸기", "`Notice.category=3` 분기 구현하기" 같은 수정은 관찰 가능한
동작을 바꾸는 것이라 리팩토링이 아니라 재설계다. 존재는 기록해 두되(`smells.md` 11, 13번), **별도
재설계 단계의 백로그**로 이관한다.

## 실행 순서 요약

```
BL-01(테스트) → BL-02(enum) ─┐
                 BL-03(DI)  ─┼→ BL-04(중복 제거) → BL-05(God Class 분해) → BL-07(캡슐화/Feature Envy)
                              │
                              └→ BL-06(DTO) → BL-08(파라미터/네이밍)

BL-09, BL-10 은 이번 범위 밖(재설계 단계 백로그로 이관)
```

모든 항목의 완료 판정은 BL-01에서 만든 특성화 테스트가 각 단계 이후에도 변경 없이 green을 유지하는지로
검증한다.
