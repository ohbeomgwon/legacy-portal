# 4-15. 리팩토링 종합 요약

`refactor/start` 태그(특성화 테스트 6개로 안전망을 확보한 시점)부터 지금까지 결재(`approval`) 도메인에
적용한 리팩토링 전체를 시간 순으로 정리한다. 개별 단계의 자세한 근거는 `docs/4-13`(상태 전이 규칙),
`4-14`(Move Method 로그), `REPORT.md`(중간 시점 요약)를 함께 참고한다.

## 0. 출발점 — 안전망 구축

- `ApprovalServiceProcessApprovalCharacterizationTest` 특성화 테스트 6개 작성
  (`@DataJpaTest` + `@Import` + `@Autowired`, 서비스 직접 `new` 금지, 데이터는 `service.create()`로만 생성)
  — 상신→승인, 반려, 취소, 권한없는 승인, 없는 id, 재승인 6개 시나리오.
- `refactor/start` 태그로 이 시점을 고정. 이후 모든 단계는 이 6개 테스트가 green을 유지하는지로 검증했다.

## 1. 이름 개선 — Rename Variable / Remove Temp

`d→approval`, `u→actor`, `s→status`로 리네임하고, `action`을 그대로 복사해 담던 임시변수 `proc`를
제거해 `action`을 직접 사용하도록 바꿨다. 동작 변화 없음.

## 2. Primitive Obsession 해소 — `status`(int) → `ApprovalStatus` enum

- `ApprovalStatus`(DRAFT=0, SUBMITTED=1, APPROVED=2, REJECTED=3, CANCELED=9) 신설.
- DB 저장값은 `ApprovalStatusConverter`(`AttributeConverter`)로 기존 정수 그대로 유지
  (`@Enumerated` 미사용 — STRING/ORDINAL 둘 다 DB값을 바꾸므로).
- API 응답의 JSON도 `@JsonValue`를 `getCode()`에 붙여 기존과 동일한 정수로 직렬화.
- `statusLabel()`의 5분기 if-else + `tmp` 임시변수를 제거하고 `ApprovalStatus.label()`에 위임
  (Move Method). "알수없음" 폴백은 타입 시스템이 잘못된 상태값 자체를 막아줘서 자연히 사라짐.

## 3. `processApproval()` 매직넘버 제거

- action 코드(1/2/3/9)를 `ApprovalStatus.Action` enum으로 매핑(처음엔 서비스 private 중첩 enum이었다가,
  "상태"와 "상태 전이 명령"이 같은 개념 축이라는 이유로 `ApprovalStatus` 안으로 이동해 한 곳에 모았다).
  `processApproval`의 public 시그니처(`int action`)는 계약 보존을 위해 그대로 유지.
- type(지출)·금액 임계값·우선순위(높음)·최소 권한 역할을 이름 있는 상수로 추출
  (`EXPENSE_TYPE_CODE`, `AUTO_PRIORITY_UPGRADE_AMOUNT_THRESHOLD`, `HIGH_PRIORITY_CODE`,
  `MIN_ROLE_CODE_FOR_APPROVAL`).
- `fromCode()`가 정의되지 않은 값에 `null`을 반환해 "정의되지 않은 action은 무시"하는 레거시 동작 보존.

## 4. God Method 분해 — Extract Method + Guard Clause

`processApproval()`의 6단계 중첩 if-지옥을 걷어냈다. 조회(대상·사용자) → 디스패치 →
`submit/approve/reject/cancel` 위임 구조로 바꾸고, 각 메서드 내부의 전제조건(상태·본인확인·권한)을
가드 클로즈 조기 반환으로 전환 — 최대 중첩 깊이 6단계 → 1~2단계.

## 5. 패키지 계층 분리

`approval` 패키지를 계층별 하위 패키지로 재구성(`git mv`로 히스토리 보존):

```
com.ktds.portal.approval
├── controller/ApprovalController.java
├── service/ApprovalService.java
├── domain/Approval.java, ApprovalStatus.java, ApprovalStatusConverter.java, AmountGrade.java
├── dto/CreateApprovalRequest.java, ProcessRequest.java, ApprovalResponse.java
└── repository/ApprovalRepository.java
```

## 6. 요청/응답 DTO 도입

`ApprovalController`가 받던 `Map<String,Object>`를 record DTO로 교체:

- `CreateApprovalRequest`(title/content/type/priority/drafterId/approverId/amount/urgent) —
  amount/urgent는 원시 타입(long/boolean)으로 선언해 필드 생략 시 레거시의
  `getOrDefault(key, 0/false)`와 동일하게 기본값 처리.
- `ProcessRequest`(userId/action/reason) — compact 생성자에서 `reason == null`이면 `""`로 정규화해
  레거시의 `getOrDefault("reason", "")` 동작 보존.
- `ApprovalResponse` — 엔티티를 직접 노출하지 않고, `status`를 `int`로 담아 JSON 계약을 그대로 유지.

엔드포인트 경로·HTTP 메서드·JSON 필드명·`status`의 정수 표현은 전혀 바뀌지 않았다(실제 앱을 기동해
curl로 생성/상신/반려 흐름을 검증).

## 7. `amountGrade()` Move Method

S/A/B/C 등급 계산 로직(임계값 1000만/100만/10만원)을 `AmountGrade` enum으로 이동. 서비스에는
`AmountGrade.fromAmount(approval.getAmount()).name()` 한 줄 위임만 남았다. `statusLabel`과 함께
"표현/도메인 규칙은 서비스의 책임이 아니다"라는 판단에 따른 이동(Feature Envy 해소).

## 8. 메일 본문 Extract Method

`approve()`/`reject()`의 메일 본문 조립을 `approvedBody()`/`rejectedBody()` private 메서드로 분리.
본문 조립 자체는 알림 문구를 다루는 서비스의 책임으로 보고 도메인으로는 옮기지 않았다.

## 9. 강결합 해소 — `MailSender`/`AuditLogger` 인터페이스 + 생성자 주입

- `SmtpMailSender`/`FileAuditLogger`를 직접 `new`하던 필드를 `MailSender`/`AuditLogger` 인터페이스로
  바꾸고 생성자 주입으로 연결(Extract Interface + DI). 구현체는 `@Component`로 등록.
- 실제로 콘솔에만 출력하는데 이름이 SMTP/File을 암시하던 Poor Naming을 바로잡아
  `SmtpMailSender→ConsoleMailSender`, `FileAuditLogger→ConsoleAuditLogger`로 리네임
  (`NoticeService`/`ScheduleService`의 참조도 컴파일 유지를 위해 기계적으로 갱신).

## 10. 상태 전이·권한 규칙을 `Approval` 도메인으로 이동

`submit/approve/reject/cancel`의 가드 클로즈(상태 확인·본인 확인·권한 판정)와 상신 시 우선순위 자동
상향 규칙을 `Approval` 엔티티의 도메인 메서드로 이동(Anemic Domain Model 해소). 레거시가 규칙 위반
시 예외 없이 조용히 무시했으므로, `Approval.submit()/approve()/reject()/cancel()`도 예외 대신
**`boolean`**을 반환한다. `ApprovalService`는 이제 "도메인 메서드 호출 → 성공 시에만 저장·메일·
감사로그"라는 오케스트레이션만 담당한다.

## 11. `AuditLogger` 계약 변경 — `write(action, id, userId)`

- 타임스탬프 포맷팅(`yyyy-MM-dd HH:mm:ss`)과 로그 라인 조립(`"[timestamp] ACTION id=.. by=.."`)이
  `ApprovalService`/`NoticeService`/`ScheduleService`에 복붙되어 있던 것을, `AuditLogger`의 계약
  자체를 `write(String action, Long id, Long userId)`로 바꿔 구현체(`ConsoleAuditLogger`) 한 곳에서만
  조립하도록 옮겼다. 출력 형식은 레거시와 동일하게 유지.
- `ApprovalService`의 `writeAudit()` private 헬퍼는 더 이상 필요 없어져 제거.
- `NoticeService`/`ScheduleService`도 인터페이스 시그니처 변경으로 호출부를 기계적으로 갱신(구조·동작
  변경 없음).
- `create()`의 감사 로그에 있던 `type=..` 부가 정보는 새 계약이 지원하지 않아 콘솔 출력에서 빠짐(DB
  저장값·API 응답과 무관한 콘솔 로그 포맷의 사소한 변화).

## 12. 신규 단위 테스트 — `FakeMailSender`

기존 안전망(특성화 테스트)은 건드리지 않고 `ApprovalServiceMailNotificationTest`를 새 파일로 추가.
Spring 빈 등록 없이 `FakeMailSender`를 테스트 코드에서 직접 `new`로 생성해
`new ApprovalService(repo, userRepo, fakeMailSender, new ConsoleAuditLogger())` 형태로 생성자에 바로
주입 — 상신/승인/반려 시 메일 발송 여부·수신자·제목을, 취소·권한없음 시 메일 미발송을 검증(5개).

## 검증 방법

- 매 단계마다 `mvn test`로 특성화 테스트 6개(+ 이후 추가된 5개, 총 11개) green 확인.
- 여러 단계(패키지 재구성, DTO 도입, 도메인 메서드 이동)에서는 실제 앱을 기동해 curl로 생성·상신·
  승인·반려·권한없는 시도 흐름을 직접 재현해 JSON 응답과 상태 전이가 레거시와 동일함을 확인.
- 새 테스트는 사용자가 명시적으로 지시했을 때만 추가(자동으로 특성화 테스트를 늘리지 않음).

## 보존된 것 (전 구간 공통)

- `public void processApproval(Long id, Long userId, int action, String reason)` 시그니처.
- API 엔드포인트 경로·HTTP 메서드·JSON 필드명.
- DB 저장값(정수) — `status` 등 enum으로 바뀐 필드도 컬럼 값은 그대로.
- 조용한 실패(대상 없음·사용자 없음·상태 불일치·권한 없음·정의되지 않은 action) — 예외 없이 무시.
- 레거시 결함(예: 상신 시 기안자 본인 확인 없음)도 "고치지" 않고 관찰 동작으로 보존.

## 아직 남은 스멜 (`docs/4-12` 백로그 기준)

- **God Class 잔존**: `ApprovalService`가 여전히 조회·저장·메일·감사로그 오케스트레이션을 한 클래스가
  담당(책임 자체는 분산됐지만 클래스는 하나).
- **`NoticeService`/`ScheduleService` 강결합·중복 미해소**: `ConsoleMailSender`/`ConsoleAuditLogger`를
  여전히 직접 `new`(리네임만 반영, DI 전환은 안 함), 감사 로그·메일 본문·상태 라벨 패턴이
  `ApprovalService`와 여전히(혹은 이제는 "우연히") 유사·중복.
- **`Approval.type`/`priority`, `User.role`**: 아직 int 그대로(enum 전환은 이번 범위 밖).
- **캡슐화 부재**: `Approval`의 모든 필드에 public setter가 여전히 열려 있음.
- **`create()` 파라미터 8개**: Long Parameter List 미해소(DTO는 컨트롤러 경계에서만 도입, 서비스
  메서드 자체는 그대로).

다음 단계는 `docs/4-12` 백로그의 BL-04(중복 제거 확장, Notice/Schedule 포함)·BL-07(캡슐화 강화) 등을
참고해 진행하면 된다.
