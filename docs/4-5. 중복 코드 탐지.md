# 4-5. 중복 코드 탐지

`approval`(결재) · `notice`(공지) · `schedule`(일정) 세 서비스(`ApprovalService`, `NoticeService`,
`ScheduleService`)에서 관찰되는 중복 코드를 유형별로 정리한다. 세 서비스는 구조적으로 "Entity 조회 →
검증/권한 판정 → 상태 변경 → 저장 → 감사 로그 → (선택) 메일 발송"이라는 동일한 흐름을 각자 따로
구현하고 있어, 로직 자체가 통째로 복붙된 경우가 많다. `NoticeService` 클래스 주석에도 명시되어 있듯
이 중복은 "공통 모듈 추출" 실습의 핵심 소재다.

## 요약 표

| 유형 | ApprovalService | NoticeService | ScheduleService |
|---|---|---|---|
| A. 감사 로그 기록 포맷 코드 | `create()` L62-65, `writeAudit()` L164-165 (부분 추출) | `create()` L44-45, `publish()` L69-70 | `create()` L62-63, `confirm()` L76-77 |
| B. 협력 객체 직접 `new` | L37-38 (mail, audit) | L24-25 (mail, audit) | L20 (audit만) |
| C. 메일 본문 생성+발송 패턴 | SUBMIT L103-107 · APPROVE L121-124 · REJECT L140-144 | `publish()` 긴급메일 L64-67 | 없음(메일 발송 로직 자체가 없음) |
| D. 조회 + null 체크 후 조용히 리턴 | `processApproval()` L76-84 | `publish()` L50-53 | `confirm()` L69-71 |
| E. 권한 판정(`role>=2`) 조건식 | APPROVE L114 · REJECT L133 (같은 클래스 내부 중복) | `publish()` L56 | 없음 |

## 상세 내용

### A. 감사 로그 기록 포맷 코드 (완전 동일 구조, 사실상 문자 그대로 복붙)

세 서비스 모두 아래 2줄 블록을 이벤트가 발생할 때마다 그대로 반복한다.

```java
String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
audit.write("[" + now + "] ACTION_NAME id=" + id + " by=" + userId);
```

- `ApprovalService.create()` (L62-65), `ApprovalService.writeAudit()` (L164-165, `processApproval()`의
  SUBMIT/APPROVE/REJECT/CANCEL 4곳에서 호출)
- `NoticeService.create()` (L44-45), `NoticeService.publish()` (L69-70)
- `ScheduleService.create()` (L62-63), `ScheduleService.confirm()` (L76-77)

`ApprovalService`는 `writeAudit()`으로 부분 추출했지만 `create()` 안에는 여전히 동일 로직이 인라인으로
남아 있어(코드 주석에도 "그나마 추출했지만 create() 안에는 또 복붙이 남아 있다"고 명시) **불완전한
중복 제거**의 예시이기도 하다. `NoticeService`·`ScheduleService`는 아예 추출 시도조차 없다.

### B. 협력 객체 직접 `new` (구조적 반복)

```java
private final SmtpMailSender mail = new SmtpMailSender();
private final FileAuditLogger audit = new FileAuditLogger();
```

`ApprovalService`(L37-38)와 `NoticeService`(L24-25)에 동일한 두 줄이 그대로 있고, `ScheduleService`는
메일 발송 기능이 없어 `audit` 한 줄만(L20) 같은 패턴으로 반복된다. 세 서비스 모두 생성자 주입 없이
구체 클래스를 직접 생성해 테스트 시 교체가 불가능하다는 동일한 결합 문제를 공유한다.

### C. 메일 본문 생성 + 발송 패턴

인사말 + 안내 문구 + 제목(+ 부가 정보) 구조의 메일 본문을 만들고 `mail.send(...)`로 보내는 코드가
`ApprovalService` 안에서만 3번(SUBMIT/APPROVE/REJECT), `NoticeService.publish()`에서 1번(긴급 공지 전체
발송 루프) 반복된다.

```java
String body = "안녕하세요 " + 수신자.getName() + "님,\n"
        + "<사건 설명>\n제목: " + 대상.getTitle() [+ "\n사유: " + reason 등];
mail.send(수신자.getEmail(), "[구분태그] " + 대상.getTitle(), body);
```

- `ApprovalService.processApproval()` SUBMIT L103-107, APPROVE L121-124, REJECT L140-144
- `NoticeService.publish()` L64-67

`ScheduleService`에는 이 패턴이 없다(메일 발송 기능 자체가 구현되어 있지 않음 — 세 서비스 간 기능
비대칭 지점).

### D. 조회 + null 체크(존재하지 않으면 조용히 리턴) 패턴

```java
X target = repo.findById(id).orElse(null);
if (target == null) return;
User user = userRepo.findById(userId).orElse(null);
if (user == null) return;
```

- `ApprovalService.processApproval()` L76-84
- `NoticeService.publish()` L50-53
- `ScheduleService.confirm()` L69-71

세 곳 모두 "대상 없음"과 "사용자 없음"을 구분하지 않고 동일하게 조용히 종료한다(호출자는 실패
사실도, 원인도 알 수 없음 — 레거시 관찰 동작으로 리팩토링 시 보존 대상).

### E. 권한 판정(`role>=2`) 조건식

```java
if (u.getRole() >= 2) {  // role 1=사원 2=팀장 3=임원
```

`ApprovalService.processApproval()`의 APPROVE(L114)와 REJECT(L133) 분기에 **같은 클래스 안에서도**
동일 조건식이 두 번 나오고, `NoticeService.publish()`(L56)에도 같은 조건식이 그대로 반복된다. "팀장
이상만 처리 가능"이라는 동일한 권한 규칙이 세 곳에 하드코딩되어 있어, 정책이 바뀌면(예: `role>=2` →
`role>=3`) 세 곳을 모두 찾아 고쳐야 하는 산탄총 수술(Shotgun Surgery) 위험이 있다.

### F. 상태 라벨 변환(`statusLabel`) if/else 사다리

```java
if (s == 0) return "...";
else if (s == 1) return "...";
...
else return "알수없음";
```

- `ApprovalService.statusLabel()` L169-179 — 0/1/2/3/9 5개 분기
- `NoticeService.statusLabel()` L76-82 — 0/1/9 3개 분기(구조는 동일, 상태 가짓수만 다름)

두 메서드는 "정수 상태값을 화면 표시용 한글 라벨로 바꾼다"는 동일한 책임을 각자 구현한 것으로, 상태
enum이 도입되면 각 enum이 스스로 라벨을 갖는 방식으로 통합할 수 있는 후보다. `ScheduleService`에는
대응하는 `statusLabel()`이 아예 없어(비대칭) 일정 화면은 상태 라벨을 어디서 만드는지 이 코드만 봐서는
알 수 없다.

## 참고 — 서비스 간 중복이 아닌 것 (구분을 위한 메모)

- `ApprovalService.processApproval()` 내부에서 5번 반복되는 `d.setStatus(X); d.setUpdatedAt(now);
  repo.save(d);` 패턴은 **같은 클래스 내부의 중복**이며, `Notice`·`Schedule` 엔티티에는 `updatedAt`
  필드 자체가 없어 서비스 간 비교 대상은 아니다.
- `ScheduleService.create()`의 시간 겹침 검증(`flag1` 루프, L40-49)은 다른 두 서비스에 대응 로직이
  없는 `schedule` 도메인 고유 규칙이라 중복 목록에 포함하지 않았다.

## 리팩토링 방향 참고 (실습 소재)

`CLAUDE.md`의 개선 방향(중복 제거, 직접 `new` → 생성자 주입, 매직넘버 → enum)에 따라 우선순위를
매기면 다음 순서가 자연스럽다.

1. **A(감사 로그)·B(직접 new)** — `AuditLogger`/`MailSender`를 인터페이스로 뽑고 각 서비스에 생성자
   주입, 로그 포맷팅은 공통 헬퍼(또는 도메인 공통 상위 클래스)로 추출.
2. **D(조회+null 체크)** — 반복되는 "찾아서 없으면 조용히 리턴" 흐름을 공통 유틸로 추출하되, **동작은
   그대로**(예외로 바꾸지 않음) 유지해야 한다.
3. **E(권한 판정)** — `role>=2` 조건을 `User` 도메인 메서드(예: `user.canApprove()`)로 이동해 한 곳에서만
   정의.
4. **F(상태 라벨)** — 상태값을 enum화한 뒤(단 DB 저장값은 기존 정수 그대로, `@Convert` 사용) 각 enum이
   자신의 라벨을 갖도록 이동.
5. **C(메일 본문)** — 공통 알림 조립 로직을 추출하되, 도메인마다 다른 문구·수신자 결정 규칙은 그대로
   보존.

모든 추출은 특성화 테스트가 리팩토링 전후 동일하게 green인지로 검증한다.
