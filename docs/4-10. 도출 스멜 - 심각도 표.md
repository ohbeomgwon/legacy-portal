# 4-10. 도출 스멜 - 심각도 표

`docs/4-4`(코드베이스 구조 파악) · `4-5`(중복 코드 탐지) · `4-7`(긴 메서드 탐지) · `4-8`(매직넘버 탐지) ·
`4-9`(강결합 탐지)에서 각각 진단한 스멜을 한 표로 모아 심각도순으로 정리한다. 전체 종합 목록은
`docs/smells.md`에도 동일 근거로 저장했다.

## 심각도 산정 기준

아래 네 가지를 함께 고려해 상대적 심각도를 매겼다(절대적 점수가 아니라 **우선순위 판단용 서열**).

1. **안전망 영향** — 이 스멜이 다른 스멜의 리팩토링 위험을 얼마나 증폭시키는가(예: 테스트 부재는 모든
   리팩토링의 위험을 키운다).
2. **파급 범위** — 몇 개 도메인·파일에 걸쳐 있는가(넓을수록 산탄총 수술 위험 ↑).
3. **관찰 가능한 위험** — 실제 크래시·오동작·원인 불명 실패로 이어질 가능성(설계 결함이라도 지금 당장
   터지지 않으면 상대적으로 낮게 봤다).
4. **은닉성** — 코드만 봐서는 놓치기 쉬운 정도(주석에만 의미가 있거나, 죽은 코드처럼 존재를 알아채기
   어려운 경우 가중).

| 2 | God Class | **치명** | `ApprovalService` 전체 | 4-4 |
| 3 | Long Method · 깊은 중첩 | 높음 | `ApprovalService.processApproval()` (86줄, 중첩 6단계) | 4-7 |

## 등급별 설명

### 치명 (Critical)

- **테스트 부재**가 1위인 이유는 그 자체로는 "버그"가 아니지만, 아래 모든 항목을 고치는 작업의
  안전망이 없다는 뜻이기 때문이다. 특성화 테스트 없이 God Class를 쪼개거나 매직넘버를 enum으로
  바꾸면 관찰 가능한 동작이 바뀌어도 알아챌 방법이 없다.
- **God Class**(`ApprovalService`)는 검증·영속화·메일·감사로그·포맷팅·권한판정을 전부 떠안고 있어,
  이 프로젝트에서 가장 많은 다른 스멜(Long Method·중복·강결합·Feature Envy)이 물리적으로 모여 있는
  진앙지다.

### 높음 (High)

- **Long Method·깊은 중첩**은 God Class의 증상 중 가장 구체적으로 측정 가능한 지표(86줄, 6단계)다.
- **조용한 실패**는 권한 없음·상태 불일치 시 예외 없이 무시하거나 `null`을 반환해, 호출자(프론트엔드·
  QA)가 실패 자체를 알 방법이 없다. 리팩토링 중 실수로 "고쳐버리기" 가장 쉬운 지점이라 특히 주의가
  필요하다(`CLAUDE.md` 보존 대상).
- **요청 검증 부재**는 `ApprovalController`가 `Map<String,Object>`를 그대로 캐스팅해, 키 철자가
  틀리거나 타입이 다르면 컴파일 타임이 아니라 실행 중 `ClassCastException`/`NullPointerException`으로
  터진다 — API 경계에서 발생해 외부 입력에 직접 노출된다.
- **매직넘버/Primitive Obsession**은 4개 도메인 전체에 퍼져 있고, `4-8` 문서에서 확인했듯 같은 숫자
  (`1`,`2`,`3`)가 필드마다 전혀 다른 의미라 오독 위험이 가장 크다.

### 중상 (Medium-High)

- **중복 코드**는 감사 로그·메일 본문·상태 라벨·권한 판정(`role>=2`)이 세 서비스에 복붙되어 있어,
  정책이 하나 바뀌면 세 곳을 다 찾아 고쳐야 하는 산탄총 수술 위험이 실제로 존재한다.

### 중간 (Medium)

- **강결합**(직접 `new`)은 테스트 불가 문제의 근본 원인이지만, 그 자체가 지금 당장 오동작을 일으키진
  않는다.
- **캡슐화 부재**는 현재는 서비스만 엔티티를 변경하고 있어 당장 문제는 없지만, 잠재적으로 어디서든
  상태를 임의로 바꿀 수 있는 위험을 안고 있다.
- **Feature Envy**는 금액 등급·우선순위 자동 상향 같은 도메인 규칙이 서비스에 있어 재사용·테스트가
  어렵지만, 동작 자체는 명확하게 관찰 가능하다.

### 중하~낮음 (Low-Medium)

- **Long Parameter List**·**Poor Naming**·**비대칭/미구현 매직값**·**Comment Smell**은 가독성·유지보수성
  문제이거나(파라미터 8개, 약어 변수) 죽은 코드에 가깝다(`Notice.category=3`처럼 정의만 있고 분기가
  없는 값). 지금 당장 오동작을 일으키지는 않지만 방치하면 신규 개발자의 오해를 유발한다.

## 리팩토링 착수 순서 제안

1. **특성화 테스트 작성** — 현재 관찰 가능한 동작(상태 전이 결과·응답·DB 값·조용한 실패 포함)을 먼저
   테스트로 고정한다. 이것 없이는 2번부터가 전부 위험하다.
2. **God Class 분해** — `ApprovalService.processApproval()`을 `submit`/`approve`/`reject`/`cancel`로
   내부 위임 분리(`public` 시그니처는 유지).
3. **중복 제거 + 강결합 해소를 동시에** — 감사 로그·메일 발송을 인터페이스로 추출해 생성자 주입하면서,
   세 서비스에 복붙된 로직을 공통 지점으로 모은다.
4. **매직넘버 → enum** — `4-8` 문서의 매핑표대로 진행하되 DB 저장값은 정수 그대로(`@Convert`).
5. 나머지(캡슐화, Feature Envy, 파라미터 목록, 네이밍)는 위 네 단계를 지나며 자연스럽게 함께 정리되는
   경우가 많아 마지막에 남은 부분만 손본다.

모든 단계는 1번에서 만든 특성화 테스트가 green을 유지하는지로 검증한다.
