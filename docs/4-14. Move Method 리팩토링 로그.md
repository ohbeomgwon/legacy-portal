# 4-14. Move Method 리팩토링 로그 — statusLabel / amountGrade

`docs/(SRP 책임 분류)` 채팅 분석에서 `ApprovalService`의 표현(프레젠테이션)·도메인 규칙 책임으로
지목된 `statusLabel()`·`amountGrade()`를 각각 값 타입(enum)으로 이동(Move Method)한 기록이다.

## 배경 — 왜 서비스의 책임이 아닌가

두 메서드는 `Approval` 인스턴스의 필드(`status`, `amount`)를 꺼내 "이 값이 화면에 어떻게 보여야
하는가"만 계산할 뿐, 결재 처리(검증·영속화·알림·감사로그)와는 무관한 순수 변환 로직이다.

- `statusLabel(Approval)` — 상태값 → 화면 표시 문자열("임시저장"/"상신"/...). 이건 **상태 그 자체가
  가진 정보**(`ApprovalStatus`가 이미 알고 있어야 할 것)를 서비스가 대신 계산해준 것 → Feature Envy.
- `amountGrade(Approval)` — 금액값 → 등급("S"/"A"/"B"/"C"). 이것도 **금액이라는 값 자체의 규칙**이지
  결재 처리 절차와는 무관하다 → Feature Envy(`docs/4-4`·`4-8`에서 이미 지적된 스멜).

## 적용한 변경

### 1) `statusLabel()` — 이미 완료됨 (이전 리팩토링에서 선반영)

`status`를 `ApprovalStatus` enum으로 전환할 때 5분기 if-else 사다리와 `tmp` 임시변수를 제거하고
`ApprovalStatus.label()`에 위임하도록 이미 바꿔둔 상태였다. 이번 요청으로 재확인만 했다.

```java
// ApprovalService.java
public String statusLabel(Approval approval) {
    return approval.getStatus().label();
}
```

### 2) `amountGrade()` — 이번에 새로 이동

**Before**

```java
public String amountGrade(Approval approval) {
    long amount = approval.getAmount();
    if (amount >= 10000000) return "S";
    else if (amount >= 1000000) return "A";
    else if (amount >= 100000) return "B";
    else return "C";
}
```

**After** — 등급 계산 로직을 신규 `AmountGrade` enum(`com.ktds.portal.approval.domain`)으로 옮기고,
서비스에는 한 줄 위임만 남겼다.

```java
// AmountGrade.java (신규)
public enum AmountGrade {
    S, A, B, C;

    private static final long S_THRESHOLD = 10_000_000L;
    private static final long A_THRESHOLD = 1_000_000L;
    private static final long B_THRESHOLD = 100_000L;

    public static AmountGrade fromAmount(long amount) {
        if (amount >= S_THRESHOLD) return S;
        else if (amount >= A_THRESHOLD) return A;
        else if (amount >= B_THRESHOLD) return B;
        return C;
    }
}

// ApprovalService.java
public String amountGrade(Approval approval) {
    return AmountGrade.fromAmount(approval.getAmount()).name();
}
```

`AmountGrade`는 DB에 저장되는 값이 아니라(결재 엔티티의 필드가 아니라 매번 금액으로부터 계산되는
파생값) `AttributeConverter`가 필요 없다. 반환 문자열은 enum 상수 이름(`name()`)을 그대로 써서
`"S"/"A"/"B"/"C"`라는 레거시 반환값과 동일하다.

## 보존된 것 (계약 검증)

- `ApprovalService.statusLabel(Approval)`, `amountGrade(Approval)`의 **public 시그니처**(파라미터·
  반환 타입 `String`)는 그대로다.
- 등급 경계값(1000만/100만/10만원)과 등급 문자열(S/A/B/C)이 동일 — 경계 케이스를 직접 대조했다:
  - `10,000,000` → S (기존 `amount >= 10000000` 그대로)
  - `9,999,999` → A
  - `999,999` → B
  - `99,999` → C
- 두 메서드는 현재 `ApprovalController`나 다른 어떤 코드에서도 호출되지 않는다(grep으로 확인 —
  API로 노출되지 않은 public 메서드). 그래서 JSON 응답 계약에 미치는 영향은 없다.

## 검증

새 테스트는 추가하지 않았다(사용자 지시: 특성화 테스트는 지시할 때만 작성). 기존
`ApprovalServiceProcessApprovalCharacterizationTest` 6개로 회귀만 확인했다.

```
Tests run: 6, Failures: 0, Errors: 0, Skipped: 0
BUILD SUCCESS
```

`statusLabel`/`amountGrade` 자체는 이 6개 테스트가 직접 검증하는 대상이 아니므로(원래 테스트되지
않던 메서드), 경계값 대조는 위 표처럼 코드 리딩으로 수동 확인했다.
